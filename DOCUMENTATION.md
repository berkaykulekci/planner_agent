## 🎯 What is this project?
This project is an AI-powered travel planning system that autonomously optimizes users' travel itineraries based on **real-time weather data** and **personal preferences** (outdoor/indoor).

Unlike traditional platforms with fixed algorithms that simply list places to visit, this project is built on an **Agentic AI** architecture. The system features an autonomous structure capable of making decisions based on current conditions, using external tools (APIs) on its own initiative, and self-correcting errors.

---

## 💡 What is the motivation?
Traditional vacation planning is highly static. You buy tickets to a city and make a list of places to see, but when the travel day arrives, a sudden storm or extreme heat can completely ruin all outdoor plans.

Our main motivation for this project is: **"To create a human-like travel advisor that doesn't just predict the weather, but dynamically makes the most logical decisions based on it."**

In doing so, rather than relying on pure rule-based coding, we aimed to experience modern AI concepts by combining the decision-making ability of an LLM (Large Language Model) with real-world interactions (Tool Calling).

---

## 🔍 What kind of problem is this? Which field does it refer to?
Technically, this project lies at the intersection of the following Computer Science and AI fields:
1. **Agentic Workflow / Multi-Agent Systems:** The *ReAct* architecture where the model sequentially applies Thought -> Action -> Observation steps.
2. **LLM Tool Calling:** The ability of Large Language Models to interact with real-world APIs and Python functions (e.g., Weather API, Scoring Algorithm).
3. **LLM-as-a-Judge:** The problem of evaluating and scoring the output generated by one AI using a completely independent second AI (Evaluator Agent).

---

## 🛠️ What problems is it solving?

### 1. Dynamic and Real-Time Planning Problem
When you ask standard ChatGPT or Claude to "Plan a trip to Istanbul," it provides a generic template without knowing the current weather. This project solves the "Blind Planning" problem by pulling real-time data with the **OpenWeatherMap API** and presenting this up-to-date information to the LLM.

### 2. Human Error and Objective Decision Problem (Python Scoring Tool)
If left entirely to an AI, the model might misinterpret data like temperature or rain. Thanks to the **Python-based Strict Scoring Tool** we integrated into the project, weather data is processed through a precise mathematical formula (e.g., Temperature 4 pts, Rain 3 pts, Wind 1.5 pts) to generate an objective score out of 10. This entirely eliminates LLM hallucinations regarding weather suitability.

### 3. Self-Reflection and Reliability
It is a difficult task to verify whether the generated final plan is actually consistent. Through our **Evaluation Framework (Evaluator Agent)**, the generated travel itinerary is passed to a second AI. This second agent looks for logical errors such as "Did it suggest outdoor activities while it is raining?" and grades the system. This minimizes the hallucination rate of the application.

### 4. Technology Stack and API Independence
To overcome cost and limit issues, the system is abstracted via LangChain and designed to run hyper-fast and flexibly with the **Groq API** integration (Llama-3.3-70b-versatile). Furthermore, it features a user-friendly frontend built with Streamlit.

## ⚙️ Technical Details & Frameworks (Detailed Technical Report)

### 1. Frameworks and Core Libraries
- **LangChain / LangGraph:** Used as the backbone for the multi-agent system. It manages the prompt templates, tool binding (`create_tool_calling_agent`), and the execution loops (`AgentExecutor`).
- **Streamlit:** Chosen for the frontend framework due to its seamless integration with Python scripts, allowing rapid deployment of a clean, interactive, and reactive web application.
- **Python (3.12+):** The core programming language used to build the logic, data processing, and custom scoring algorithms.

### 2. Large Language Models (LLMs)
- **Groq API (Llama-3.3-70b-versatile):** Selected over OpenAI/Gemini free tiers to prevent severe rate-limiting issues. Groq provides ultra-low latency inference using LPU (Language Processing Unit) architecture, which is critical when multiple agents are running sequentially (Planner -> Evaluator).

### 3. APIs and Datasets
Instead of using static, outdated CSV datasets for weather, the system is strictly API-driven to ensure real-world applicability:
- **OpenWeatherMap 5-Day/3-Hour Forecast API:** This acts as our primary dynamic dataset. The API feeds the agent with real-time JSON data regarding temperature, humidity, wind speeds, and precipitation for the chosen city over the requested dates.
- **Local JSON State Memory (`user_memory.json`):** A lightweight local storage solution used to persist recent user searches and UI preferences without the overhead of a full database.

### 4. System Architecture
1. **Input Layer:** The user provides a target city, a date range, and an activity preference via the Streamlit UI.
2. **Planner Agent (ReAct Loop):** 
   - *Thought:* Determines that it needs weather data.
   - *Action:* Invokes the `get_weather` tool.
   - *Observation:* Saves data to local memory state.
   - *Action:* Invokes the `score_days` Python tool, which runs the mathematical calculation without LLM hallucination.
   - *Final Answer:* Generates the structured day-by-day itinerary.
3. **Evaluation Agent (LLM-as-a-Judge):** Reviews the final output of the Planner Agent, cross-references it with the actual weather data, and outputs a strict 0-10 float score alongside a feedback report.
